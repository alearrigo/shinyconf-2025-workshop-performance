Profiling - Tools in R

Profiling is a technique used to identify performance bottlenecks in your code.

{profvis}

An interactive tool that provides a detailed visualization of your code‚Äôs execution time.

  • Installation:
install.packages("profvis")
  • Basic usage:
library(profvis)
profvis({
# Code to be profiled
})

shiny.tictoc

A tool that uses JavaScript to measure the time taken by actions in the app from the browser‚Äôs perspective.

It is very easy to add to a Shiny app:

tags$script(
    src = "https://cdn.jsdelivr.net/gh/Appsilon/shiny.tictoc@v0.2.0/shiny-tic-toc.min.js"
)
  • If you‚Äôre unfamiliar with adding JavaScript: Packaging JavaScript code for Shiny

Run any of these operations in the JavaScript console.

// Print out all measurements
showAllMeasurements()

// To download all measurements as a CSV file
exportMeasurements()

// To print out summarised measurements (slowest rendering output, slowest server computation)
showSummarisedMeasurements()

// To export an html file that visualizes measurements on a timeline
await exportHtmlReport()

Many browsers have developer tools where you can find a profiler while your app is running.

Using profvis

Locate the tool in RStudio.

The R console will display the ‚ÄúStop profiling‚Äù button. This means the profiler is active.

Run your Shiny app and interact with it. Then, you can stop the app and the profiler.

The RStudio source panel will display a new view.

The upper section profiles each line of code, while the lower section displays a FlameGraph, indicating the time required for each operation.

You can also access the ‚ÄúData‚Äù tab.

This section shows how much time and memory each operation requires, providing a summary of the measurement.

For a more in-depth review of {profvis}, you can refer to the official documentation:

  • Examples: https://profvis.r-lib.org/articles/rstudio.html
  • Integration with RStudio: https://profvis.r-lib.org/articles/rstudio.html

Pre-procesar c√°lculos

Dif√≠cil de usar si se requiere calcular en vivo, real-time (stock exchange, streaming data), o la data no puede ser guardada en cualquier lugar (seguridad, privacidad).

Estrategias de Cach√©

bindCache() nos permite guardar c√≥mputos al vuelo en base a ciertas keys.

server <- function(input, output, session) {
  filtered <- reactive({
    survey |> 
      filter(region == input$region) |> 
      filter(age <= input$age)
  }) |> 
    bindCache(input$region, input$age) |> 
    bindEvent(input$compute, ignoreNULL = FALSE)
}

Cuando una combinaci√≥n vuelva a aparecer, se leer√° el valor en lugar de recalcularlo.

bindEvent() no es obligatorio para usar bindCache().

  1. Niveles de cach√©:
  • Nivel aplicaci√≥n: cache = "app" (default)
  • Nivel sesi√≥n: cache = "session"
  • Personalizado: cache object + opciones
bindCache(..., cache = "app")

Por defecto, se usar√° un m√°ximo de 200 Mb de cach√©.

Potencialmente mucho uso de memoria/storage, para reducir tiempo de procesamiento.

Comunicaci√≥n servidor - navegador

  • Enviar datos desde el servidor hacia el navegador toma tiempo.
    • Datos grandes -> mayor tiempo de env√≠o.
    • Conexi√≥n lenta -> mayor tiempo de env√≠o
  • Igualmente, si los datos son grandes, al navegador le cuesta m√°s leerlos y mostrarlos al usuario. Puede que la PC del usuario sea una tostadora!
  • ¬øQu√© hacer?
    • Reducir frecuencia de env√≠os (bindEvent())
    • Reducir tama√±o de env√≠os
    • Mandar lo mismo pero en partes m√°s peque√±as (server-side processing o streaming)

Reducir tama√±o de env√≠os

Es posible delegar ciertos c√°lculos al navegador. Por ejemplo, renderizar un gr√°fico con {plotly} en lugar de {ggplot2}.

plotly::plotlyOutput() # en lugar de plotOutput
plotly::renderPlotly() # en lugar de renderPlot

Con ello, se manda la ‚Äúreceta‚Äù del gr√°fico en lugar del gr√°fico mismo. Al recibir la receta, el navegador se encarga de renderizarla.

Estas funciones traducen sintaxis de ggplot2 a sintaxis de plotly.js de manera bastante eficiente. Tiene soporte para muchos tipos de gr√°ficos.

Pero no te conf√≠es, en muchos casos, el c√≥digo va a necesitar retoques. Especialmente al usar extensiones de ggplot2.

Optimizaci√≥n - Async

Programaci√≥n s√≠ncrona

  • Las tareas se ejecutan secuencialmente
  • Es f√°cil de entender e implementa

Ejemplo: Una cocina con una hornilla. Si empec√© a freir pollo, no puedo freir nada hasta terminar de freir el pollo.

Proramaci√≥n as√≠ncrona

  • Las tareas pueden iniciar y ejecutarse independientemente
  • Mientras se procesa una tarea, otras pueden ser iniciadas o completadas.

Ejemplo: Una cocina con m√∫ltiples hornillas. Si empec√© a freir pollo en una hornilla, puedo freir otra cosa en una hornilla diferente.

Hornilla == Proceso en la PC

Cuidado

A m√°s hornillas, tambi√©n es m√°s f√°cil quemar la comida!

Posibles complicaciones

  • El c√≥digo se hace m√°s dificil de entender
  • Sin adecuado control, pueden sobreescribir sus resultados
  • L√≥gica circular. Proceso A espera a Proceso B, quien espera a Proceso A
  • Incrementa la dificultad para hacer debugging porque los errores ocurren en otro lado
  • Mayor consumo de energ√≠a

Beneficios

  • Operaciones largas no bloquean a otras operaciones
  • Flexibilidad: mi sistema se adapta a retrasos inesperados
  • La aplicaci√≥n se mantiene responsiva, no se ‚Äúcuelga‚Äù
  • Uso eficiente de recursos. ‚ÄúPagu√© por 8 procesadores y voy a usar 8 procesadores‚Äù.
  • Escalabilidad a otro nivel

Casos de uso

  • Operaciones I/O:
    • Query a bases de datos
    • Solicitudes a APIs
  • C√°lculos intensivos

¬øQu√© necesito?

  • CPU con varios n√∫cleos/hilos.
  • Paquetes:
    • {promises}
    • {future}
    • ExtendedTask (Shiny 1.8.1+)

ExtendedTask es un recurso bastante nuevo. Tambi√©n es posible usar solo future() o future_promise() dentro de un reactive para lograr un efecto similar, aunque con menos ventajas.
